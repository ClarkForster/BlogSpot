# 写技术文章时，我如何组织“解释”的顺序

解释一件复杂的事，最怕的不是写得少，而是写得没有路径。

很多文章的问题并不在于内容错误，而在于它默认读者已经知道作者脑中的地图。结果就是：每一段都能看懂，但整篇文章看完以后还是不知道为什么这样安排。

## 我常用的顺序

写技术文章时，我会尽量按下面的节奏展开：

1. 先交代问题是什么。
2. 再说明为什么直觉解法不够好。
3. 然后引入真正的模型或机制。
4. 最后才讨论实现细节、边界条件和例外。

这个顺序的核心是：先帮读者建立“为什么”，再进入“是什么”和“怎么做”。

## 解释不是压缩，而是重建

很多人把写作理解成信息压缩，我更愿意把它看成认知重建。

你不是把脑中的结论倒出来，而是替读者重走一次推理路径。好的文章会让人感觉：即使第一次接触这个主题，也能自然地被带到正确的位置。
